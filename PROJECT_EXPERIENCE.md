## TicketFlow 高并发票务系统

### 项目概述

从零独立设计并开发了一款面向大型演唱会场景的高并发票务秒杀系统，涵盖用户端选座购票、商家端活动管理、管理端权限控制三大角色，完整打通从活动发布、座位管理、秒杀抢票、支付宝沙箱支付到出票核销的全业务闭环。

### 技术栈

- **后端**：Spring Boot 3.5 + Spring Security + JWT + MyBatis-Plus 3.5 + Redisson + Redis
- **前端**：Vue 3 + Vite + Element Plus + Pinia + QRCode
- **数据库**：MySQL 8.x + Redis
- **支付**：支付宝沙箱 Native 支付（RSA2 签名 + 异步通知回调）
- **文档**：OpenAPI 3.0（YAML 格式手写完整接口规范）
- **工具链**：Maven、Git、Knife4j、Hutool

### 核心功能与技术实现

**1. 秒杀抢票引擎**

针对高并发抢票场景，设计了多级防超卖架构：
- **Redis 预热库存**：活动开售时将库存缓存至 Redis，秒杀时原子 DECR 扣减，毫秒级响应
- **Redisson 分布式锁**：对同一活动的并发请求加锁排队，锁内二次校验防止并发重复购买
- **乐观锁占座**：DB 层面通过 `UPDATE seat SET status=1 WHERE id=? AND status=0` 防止座位超卖
- **双重去重**：Redis Set 快速判断用户是否已购买 + DB 权威兜底验证（仅检查已支付订单），防止 Redis 脏数据误拦
- **异步订单创建**：秒杀接口快速返回订单号，订单落库交由线程池异步执行，保证接口吞吐量
- **支付超时自动释放**：订单创建 10 分钟内未支付自动作废，释放座位和库存，保证资源有效利用

**2. RBAC 权限 + 多租户**

- 用户 → 角色 → 权限 的三级权限模型，支持 SUPER_ADMIN 全局绕过
- Spring Security + JWT 无状态认证，每次请求从 DB 实时查询用户权限
- 前端路由级权限守卫 + 后端方法级 `@PreAuthorize` 双重控制
- 多租户数据隔离：ThreadLocal + HandlerInterceptor 传递租户上下文

**3. 支付宝沙箱 Native 支付**

- 手动实现支付宝 `alipay.trade.precreate` 接口的 RSA2 签名（SHA256withRSA）
- 异步通知回调接口的验签处理（含开发环境降级策略）
- 前端轮询支付状态 + 支付成功自动跳转订单页
- 利用 serveo.net 内网穿透使得本地开发环境能接收支付宝回调

**4. 前端 SVG 交互式选座**

- 纯 SVG 绘制扇形场馆座位图，支持 VIP/内场/看台三区域分区渲染
- 座位实时状态区分（可选/已选/已售/锁定），点击切换选座
- 选座后一键秒杀 → 二维码弹窗 → 支付状态自动检测 → 跳转订单页，完整闭环

**5. 其他功能**

- 活动/商家/座位/票务完整 CRUD
- 数据统计面板（订单数、销售额、出票量）
- 字典管理、文件上传
- 完整的 OpenAPI 3.0 接口文档

### 项目难点与解决方案

| 难点 | 解决方案 |
|------|---------|
| 高并发下座位超卖 | 乐观锁 + 分布式锁 + Redis 原子扣库存，三级防护 |
| Redis 数据与 DB 不一致 | Redis 快速路径 + DB 权威兜底，不一致时清除 Redis 脏数据 |
| Snowflake ID 到前端精度丢失 | Jackson 自定义序列化 Long → String，取消/支付改用 orderNo |
| 支付宝签名验签 | 排查网关验签字符串匹配，确认 sign_type 参与签名 |
| 支付状态无法及时感知 | 前端 3 秒轮询 + 支付宝异步回调双通道确认 |
| 支付超时资源浪费 | 10 分钟定时作废机制，查询/操作前自动触发 |

### 项目经验总结

通过本项目深入理解了高并发场景下的核心技术挑战：从 Redis 分布式锁的选型与调优、MySQL 乐观锁与事务边界的处理、异步化对用户体验的提升，到前后端联调中精度、状态同步、安全控制的细节处理。独立完成从 0 到 1 的全栈开发，积累了完整的中型系统架构设计经验。
